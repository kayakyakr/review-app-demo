Review Apps Postmortem

Drawbacks

Fragmented environment
  • Sometimes it's confusing for internal testers to know what features are active
  • Ex: If there's a bug fixed in one branch, but still present in another
  • Will probably always be difficult to get product owners to understand how to test w/ review apps
Empty database
  • Gives you a fresh start, but means you need to recreate all testing objects.
  • The proper way to load a DB is via a seed file.
  • Either the seed file needs to be large, or you need to recreate everything leading up to test case
  • Could copy staging database over, but there are no fixed implementations
  • You also wind up with too much stale data.
  • Copying db would allow testing the migration path, though.
Client buy-in
  • It's just another thing to convince clients to use to get most out of it
3rd party things
  • Some 3rd parties want fixed URLS, makes it hard to do things like:
  • Oauth
  • Extensions
  • Etc
