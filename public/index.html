Some other thoughts

Working Pivotal flow
  • Ticket is delivered when PR is created
  • PM (or client) uses Pivotal's QA reviews to pass or fail ticket
  • When ticket is passed, developer merges to master (deploys to staging) and moves ticket above a line
  • Active clients can accept/reject from staging (rejections should be rare because of the reivew)
  • Production deploys can happen any time there are no unaccepted tickets above the line
  • Note: Detached clients make this flow hard. In GP, we had to mark everything as accepted ourselves after weekly demo/retro.
